# FlowDeck gotchas

- **api-zod index must be a single re-export.** `lib/api-zod/src/index.ts` must be ONLY `export * from "./generated/api";`. Merges/regeneration sometimes reintroduce extra exports that cause a TS2308 duplicate-export error (e.g. `ListTasksParams`). If `typecheck:libs` fails with TS2308 after codegen or a merge, restore it to the single line.
  **Why:** orval generates a flat namespace; duplicate barrel exports collide.
  **How to apply:** check this file right after running `pnpm --filter @workspace/api-spec run codegen` or resolving a merge.

- **api-server does NOT hot-reload route logic.** After editing anything under `artifacts/api-server/src` (routes, lib), you must restart the `artifacts/api-server: API Server` workflow for changes to take effect. Verifying via curl without restarting tests stale code.

- **orval codegen transiently wipes `generated/`.** During `codegen`, the generated dir is emptied mid-run, so vite/metro may log "Unable to resolve ./generated/api" briefly. These errors are transient — ignore them once codegen completes.

- **Baseline typecheck noise.** A full `pnpm run typecheck` fails on pre-existing `artifacts/flowdeck/src/components/ui/calendar.tsx` and `ui/spinner.tsx` due to React 19 duplicate `@types/react` resolution. This is baseline, unrelated to your changes. Verify only that YOUR changed files are error-free, e.g. `npx tsc --noEmit -p tsconfig.json 2>&1 | rg "error TS" | rg -o "^[^(]+\.tsx?" | sort | uniq -c`.

- **Drizzle transactions + helpers.** To log activity atomically with a task mutation, pass the transaction executor through helpers. The reusable executor type is `DbExecutor` exported from `artifacts/api-server/src/lib/activity.ts` (`typeof db | tx`); type helper params as `DbExecutor` so they work both inside and outside a `db.transaction`.

- **Project list endpoint is admin-only.** `GET /api/projects` returns 403 for normal users. To fetch a user's tasks for testing, hit `GET /api/projects/:projectId/tasks` directly (seeded user teste@user.com owns projects 9-12).

- **Date-only fields: never `new Date("YYYY-MM-DD")`.** Task `dueDate` (and similar) are date-only strings; `new Date(str)` parses as UTC and shifts the day back one in negative timezones. Use `parseDueDate` from `artifacts/flowdeck/src/lib/date.ts` (splits the string into local Y/M/D) for any due-date rendering/positioning. The original Roadmap view predates this helper.
  **Why:** UTC parsing causes off-by-one day bugs in calendar/timeline/table views.
  **How to apply:** import from `@/lib/date` whenever displaying or comparing a `YYYY-MM-DD` value in the flowdeck web app.

- **Task-detail sheet shows stale data on reopen unless you write the mutation RESPONSE to its query cache.** The TaskSheet is conditionally rendered (`{openTaskId && <TaskSheet/>}`) so it UNMOUNTS on close, which cancels the in-flight invalidation refetch of `useGetTask` (enabled:open, default staleTime 0). Therefore `invalidateQueries(getGetTaskQueryKey(id))` ALONE leaves the pre-edit value cached, and reopening shows old data until a second close/reopen. Fix: in every task-editing mutation's onSuccess do `queryClient.setQueryData(getGetTaskQueryKey(taskId), updated)` using the PATCH response (`PATCH /tasks/:id` returns the full `Task`). Covers `saveField` (title/type/dueDate/completed/columnId/priority/assigneeId/labelIds/mindmapId), the floating-save title path, mindmap link, and the description editor.
  **Why:** unmount-on-close cancels the refetch, so invalidate-only never updates the cache before the next open; the server response is authoritative and timing-independent.
  **How to apply:** never rely on invalidate-only to refresh the task-detail sheet — setQueryData with the mutation response. Same principle underlies `flowdeck-rich-text-editor.md`.

- **Mindmap editor canvas coordinates run under a CSS `scale(zoom)` layer.** The canvas has an outer `canvasRef` div (which holds the pointer move/up/leave handlers) wrapping an inner "scale layer" (`absolute inset-0 origin-top-left` with `transform: scale(zoom)`) that contains areas, the pointer-events-none SVG edge layer, and nodes. Convert pointer→canvas coords with `(clientX - rect.left) / zoom` (rect = `canvasRef.getBoundingClientRect()`); node/area `x/y` are stored in canvas (unscaled) space. To detect a click on empty background (pan/marquee), attach `onPointerDown` to the OUTER `canvasRef` container (NOT the scale layer) and guard with `e.target === e.currentTarget || e.target === contentRef.current` (contentRef = the scale layer). Reason it must be the outer container: at zoom < 1 the scale layer shrinks toward the top-left and no longer covers the viewport, so clicks in the empty margin land on the outer container — if the handler lived on the scale layer those clicks would be dead. node/area handlers do NOT `stopPropagation`, so they bubble to the container; the guard rejects them because target is the node/area, not the container or scale layer. For marquee node hit-testing, compare each node's `getBoundingClientRect()` against the marquee rect in CLIENT space (exact, zoom-agnostic); render the marquee rect itself in canvas coords inside the scale layer so it scales with zoom.
  **Why:** the zoom transform means canvas-space math and client-space math are off by the zoom factor; mixing them silently misplaces drags/selection, DOM rects are the only zoom-accurate node bounds, and a scaled layer does not fill the viewport at low zoom.
  **How to apply:** any new pointer interaction in `artifacts/flowdeck/src/pages/mindmap-editor.tsx` — divide by `zoom` for canvas coords, put background pointer-down on the outer container guarded by `target===currentTarget||target===contentRef.current`, and hit-test via `getBoundingClientRect` in client space.

- **Mindmap navigation gesture model (drag-empty = PAN).** A plain left-drag on empty canvas PANS; marquee multi-select moved to Shift+left-drag. Also: middle-mouse drag and Space+left-drag pan from anywhere (even over nodes) via a `panGesture(e)` guard at the top of the node/area/resize pointer-down handlers; wheel pans vertically, Shift+wheel horizontally (plain mouse wheels emit only deltaY, so horizontal needs Shift+wheel or drag), Ctrl/Cmd+wheel zooms to cursor. A `"pan"` DragState is handled first in onPointerMove (return early) and excluded from the `dragMoved ⇒ setDirty` branch in onPointerUp so panning never marks the map dirty. `fitView()` frames all `[data-node-id],[data-area-id]` via client rects → world coords.
  **Why:** the original build only panned via `deltaX`, which real mouse wheels never emit, so horizontal pan and reaching off-center regions were impossible.
  **How to apply:** keep pan the default empty-drag; if adding gestures, route middle/space pans through `panGesture`, and never let a pan set `dirty`.

- **Unsaved-changes navigation guard is per-link, not global (wouter has no built-in blocker).** The pattern lives in `artifacts/flowdeck/src/lib/unsaved-guard.tsx` (`UnsavedGuardProvider` + `useUnsavedGuard`), wired in `App.tsx` inside `WouterRouter`. A dirty editor calls `register({ isDirty, save: () => Promise<void>, discard })` in an effect (and `register(null)` on unmount); navigation entry points call `guardedNavigate(to)`, which shows an AlertDialog (Salvar e sair / Descartar / Cancelar) only when the registered handler is dirty. Because wouter can't intercept navigation centrally, EVERY way to leave a dirty editor must route through `guardedNavigate` OR `beforeunload` — currently the sidebar `NavLink` (shell.tsx) and the editor back button use `guardedNavigate`, and the editor adds a `window` `beforeunload` (native prompt) while dirty. Logout and the mindmap-to-mindmap `openReferencedMap` path intentionally bypass the dialog (openReferencedMap auto-saves first). `save` must be an awaitable that reads `latestRef.current` (not stale closure state) so the guard registration stays stable.
  **Why:** wouter `<Link>`/`setLocation` navigate synchronously with no blocker hook, so a browser-native `beforeunload` covers tab-close/reload but in-app links need explicit interception at each entry point; missing one silently drops unsaved edits.
  **How to apply:** when adding a new navigation control that can leave a dirty editor (new sidebar item, breadcrumb, header link), route it through `guardedNavigate`; when giving another editor (e.g. note-editor) the same protection, have it `register` a handler and add its own `beforeunload`.

- **Mindmap drag needs pointer capture once negative coords are allowed.** The canvas container holds `onPointerMove/Up/Leave` and ends any drag on `onPointerLeave`. That was fine while drag positions were clamped with `Math.max(0,...)`, but once you allow dragging a node/area up/left into negative coordinates, a drag that leaves the container edge would terminate early. Fix: on the first real move (when `dragMoved` flips true) call `canvasRef.current.setPointerCapture(e.pointerId)` so move/up keep firing off-canvas; keep `onPointerLeave` as a pre-capture fallback and add `onPointerCancel` for interrupted drags. Do NOT also wire `onLostPointerCapture` to the same handler — the browser auto-releases capture right after `pointerup`, so it would double-invoke the end-drag logic.
  **Why:** without capture the drag stops at the viewport edge exactly in the up/left direction the clamp-removal was meant to enable.
  **How to apply:** any new drag interaction on the mindmap canvas should rely on pointer capture at drag start, not on the pointer staying inside the container.

- **Per-user rows still need workspace scoping on read.** Anything keyed only by `userId` (e.g. notifications) leaks across tenants once a user is removed from a workspace: the old rows persist and would expose task/project metadata. Any list/count endpoint that returns task/project details must join task→project and filter `inArray(projects.workspaceId, getAccessibleWorkspaceIds(user))` (in `artifacts/api-server/src/lib/access.ts`: admin=all, else owned ∪ member workspace ids), returning empty when the user has no accessible workspaces. Lazy generators (e.g. `ensureDueSoonNotifications`) must scope the same way.
  **Why:** removal from a workspace deletes membership but not historical per-user rows; scoping by userId alone is a multi-tenant data leak (architect-flagged).
  **How to apply:** when adding any user-scoped endpoint that surfaces workspace-owned content, gate it through `getAccessibleWorkspaceIds`, not just the recipient id. Mutations that return no metadata (mark-read/read-all) don't need it.
